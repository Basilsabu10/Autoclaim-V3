Slide 2: Introduction

AutoClaim is a full-stack, AI-driven motor insurance claim management platform designed to modernise the traditional claims workflow. Insurance companies today handle claim submissions manually, resulting in high fraud risk, slow turnaround, and costly human error.

AutoClaim replaces that process by:
- Providing a three-role web portal (Policyholder, Agent, Administrator) for end-to-end digital claim management.
- Applying a seven-stage AI analysis pipeline to every submitted claim: Perceptual image hashing → EXIF metadata extraction → Error Level Analysis (ELA) for image manipulation detection → Groq Vision OCR number plate recognition (with EasyOCR fallback) → YOLOv11 instance segmentation for damage & parts detection → Groq LLaMA Vision for forensic/fraud analysis → a 28-rule deterministic verification engine that outputs a final recommendation (APPROVE / REVIEW / REJECT).
- Delivering real-time notifications, analytics dashboards, and a repair cost estimator with USD-to-INR conversion.

The system is built with a React 19 (JavaScript/JSX) (Vite) frontend and a FastAPI + Python backend, using SQLite via SQLAlchemy ORM. The AI stack is completely self-hosted (YOLOv11) and API-driven (Groq), eliminating the need for expensive proprietary insurance AI tools.

Slide 3: Problem Statement

Current Challenges in Motor Insurance Claims:

Problem Impact
Manual document verification takes days to weeks Policyholder dissatisfaction, delayed payouts
Insurance fraud (staged accidents, edited images, recycled photos) Indian insurers lose ~₹30,000 crore/year to fraud
Subjective damage assessment by individual surveyors Inconsistent payout amounts; disputes
No automated license plate / vehicle match Easy to submit claims for a different vehicle
No pre-existing damage detection Policyholders claim for prior damage
Opaque decision-making Claimants cannot understand why claims are rejected

Core Problem: The insurance industry lacks an affordable, AI-integrated platform that can automate:
1. Identity verification of the vehicle (plate OCR + YOLO)
2. Damage assessment (severity, affected parts)
3. Fraud detection (image forensics, EXIF metadata, duplication checks)
4. End-to-end claim lifecycle management with role-based access

Slide 4: Project Objectives

  1. Automate claim intake — Allow policyholders to submit claims with damage photos and receive an immediate AI-generated analysis.
  2. AI Damage Assessment — Deploy YOLOv11 instance segmentation to detect & classify vehicle damage (dents, scratches, cracks, shatter, missing parts) and identify affected car panels.
  3. Forensic Fraud Detection — Use Groq LLaMA Vision to extract forensic indicators: screen recapture flags, watermarks, inconsistent lighting, pre-existing rust, etc.
  4. Vehicle Identity Verification — Match OCR-extracted license plate against the policy registration; cross-check vehicle make/model/color.
  5. Rule-Based Decision Engine — Apply 16 deterministic rules across 5 phases (Integrity, Vehicle/Damage, Contextual, Financial, Policy/History) to produce APPROVE / FLAGGED / REJECT verdicts with an auditable score.
  6. Cost Estimation — Generate a part-by-part INR repair cost breakdown (mapped from USD industry data at ₹84/USD).
  7. Role-Based Dashboard — Provide separate portals for Policyholders, Claims Agents, and Administrators with claim assignment, internal notes, bulk actions, and analytics.
  8. Real-Time Notifications — Notify policyholders in-app when their claim status changes.

Slide 5: Purpose and Need of the Project

Why AutoClaim is Needed:

Business Need:
- Indian motor insurance market processes millions of claims per year; insurers spend heavily on manual survey and fraud investigation.
- Claim settlement ratio in India averages 70-80%, leaving many valid claims unresolved due to backlogs.

Technological Need:
- Modern computer vision (YOLOv11) and multimodal LLMs (Groq/LLaMA) are mature enough for reliable vehicle damage analysis.
- No affordable open-source platform currently integrates OCR + YOLO + LLM + rule engine into a single insurance workflow.

Social Need:
- Faster, transparent, and fair claim settlement improves policyholder trust.
- Automated fraud detection protects honest policyholders from inflated premiums caused by fraudulent claims.

Academic Need:
- Demonstrates applied deep learning (YOLOv11 instance segmentation), LLM-based document intelligence, and full-stack web engineering in a real-world domain.

Slide 6: Scope of the Project

In Scope:

  • Motor vehicle insurance claims only (not life, health, or property insurance)
  • Web application (desktop browser; responsive UI)
  • Three user roles: Policyholder (User), Claims Agent, Administrator
  • AI pipeline: EXIF + OCR + YOLO + Groq + Rule Engine + Cost Estimator
  • Claim lifecycle: Submission → AI Analysis → Agent Review → Approval/Rejection → Notification
  • Analytics: Admin dashboard with fraud distribution, approval rate, cost averages, daily trend charts
  • Policy management: Admins can create policy plans; users are linked to policies
  • Document management: Supplementary document uploads after initial submission

Out of Scope:

  • Actual insurance payouts or payment gateway integration
  • Mobile application (iOS / Android)
  • Third-party API for live auto parts pricing (internal price table is used)
  • Multi-language support
  • Integration with external insurance management systems (e.g., Duck Creek, Guidewire)

Slide 13: Database Design

AutoClaim uses SQLite (autoclaim.db) via SQLAlchemy ORM. The schema has 8 tables:

Entity Relationship Summary

users (1) ──< claims (many) — via user_id
users (1) ──< policies (many) — via user_id
policies (many) >── policy_plans (1) — via plan_id
claims (1) ──── forensic_analyses (1) — one-to-one
claims (1) ──< claim_notes (many) — via claim_id
claims (1) ──< claim_documents (many) — via claim_id
claims (1) ──< notifications (many) — via claim_id
users (agent) (1) ──< claims (assigned) — via assigned_agent_id

Table Definitions

users — Accounts for policyholders, agents, admins
- id, email (unique), hashed_password, role (user/agent/admin), name, policy_id, vehicle_number, created_at

policy_plans — Insurance product templates
- id, name, description, coverage_amount, premium_monthly, created_at

policies — Active user policies
- id, user_id→users, plan_id→policy_plans, vehicle_make, vehicle_model, vehicle_year, vehicle_registration, start_date, end_date, status (active/expired/cancelled), created_at

claims — Insurance claims
- id, user_id→users, policy_id→policies, assigned_agent_id→users, description, image_paths (JSON), front_image_path, estimate_bill_path, status (pending/processing/approved/rejected/failed), accident_date, vehicle_number_plate, ai_recommendation, estimated_cost_min, estimated_cost_max, created_at

forensic_analyses — Full AI analysis for each claim (one-to-one with claims)
- EXIF fields: exif_timestamp, exif_gps_lat, exif_gps_lon, exif_location_name, exif_camera_make, exif_camera_model
- OCR fields: ocr_plate_text, ocr_plate_confidence, ocr_raw_texts
- YOLO fields: yolo_damage_detected, yolo_detections, yolo_severity, yolo_summary
- Forensic flags: is_screen_recapture, has_ui_elements, has_watermarks, image_quality, is_blurry
- Vehicle identity: vehicle_make, vehicle_model, vehicle_year, vehicle_color, license_plate_text, license_plate_match_status
- Damage: ai_damage_type, ai_severity, ai_damaged_panels, ai_structural_damage, damage_severity_score, impact_point, paint_damage, glass_damage, is_rust_present, airbags_deployed, fluid_leaks_visible, parts_missing
- Cost: ai_cost_min, ai_cost_max, repair_cost_breakdown (JSON)
- Pre-existing: pre_existing_damage_detected, pre_existing_indicators, pre_existing_confidence
- Risk: ai_risk_flags (JSON), fraud_probability, fraud_score
- Decision: overall_confidence_score, ai_recommendation, ai_reasoning, human_review_priority
- Meta: ai_raw_response, ai_provider, ai_model, analyzed_at, analysis_version

notifications — In-app alerts for claim status changes
- id, user_id→users, claim_id→claims, message, is_read, created_at

claim_notes — Agent/admin internal notes
- id, claim_id→claims, author_id→users, note, created_at

claim_documents — Supplementary files uploaded post-submission
- id, claim_id→claims, uploaded_by_id→users, file_path, label, created_at

Algorithm 1 — 7-Stage AI Analysis Pipeline (Slide 20)

FUNCTION analyze_claim(damage_images, front_image, description):

  result ← initialize empty result dict

  // Stage 1: Perceptual Image Hashing
  image_hashes ← hash_claim_images(all_images)   // pHash via imagehash
  result.metadata.image_hashes ← image_hashes

  // Stage 2: EXIF Metadata Extraction
  FOR each image IN damage_images:
    exif ← extract_exif(image)       // Read EXIF via Pillow
    IF exif has timestamp OR gps:
      result.metadata ← exif
      BREAK

  // Stage 3: Error Level Analysis (ELA)
  ela_result ← analyze_claim_images_ela(damage_images)
  IF ela_result.editing_detected:
    Mark as MANIPULATED OR GENERATED IMAGE
    Boost fraud score to ≥ 0.8

  // Stage 4: OCR — Groq Vision (primary) + EasyOCR (fallback)
  IF front_image exists:
    result.ocr ← groq_vision_extract(front_image)   // Llama-4-Scout
    IF groq_result is None:
      result.ocr ← easyocr_extract(front_image)      // Fallback
  IF ocr confidence < 0.5 AND YOLO is available:
    plate_crop ← yolo_find_plate_region(all_images)
    IF plate_crop found:
      result.ocr ← extract_number_plate(plate_crop)  // Higher accuracy on crop

  // Stage 5: YOLOv11 Instance Segmentation
  FOR each image IN damage_images + [front_image]:
    seg ← yolo11m_seg.detect(image)
    IF seg.success:
      result.yolo_damage ← seg
      BREAK

  // Stage 6: Groq LLaMA Vision — Forensic & Fraud
  extraction ← build_defaults_from_yolo(result.yolo_damage, result.ocr)
  IF groq_available:
    groq_data ← groq.extract(all_images, description)
    IF groq_data.success:
      extraction.forensics ← groq_data.forensics
      extraction.identity  ← groq_data.identity
      extraction.fraud     ← groq_data.fraud_analysis
  // Integrate ELA results into forensics
  extraction.forensics.ela_score ← ela_result.highest_ela_score

  // Stage 7: 28-Rule Verification Engine
  vdata ← prepare_verification_input(extraction, result.metadata, result.ocr)
  engine ← VerificationRules()
  verdict ← engine.verify_claim(claim_amount, vdata, policy_data, history)
  result.verification ← verdict
  result.ai_analysis.recommendation ← verdict.status   // APPROVED/FLAGGED/REJECTED

  RETURN result
END FUNCTION

Algorithm 2 — 28-Rule Verification Engine (Slide 21)

The engine runs 5 phases in sequence. Each failed rule contributes a weighted score. The final verdict is determined by the score and any CRITICAL failures.

SCORE WEIGHTS: CRITICAL=10, HIGH=5, MEDIUM=2, LOW=1
REJECT if score ≥ 10 OR any CRITICAL failure
FLAG   if score ≥ 2
APPROVE otherwise

PHASE A — Integrity & Source Checks:
  Rule 1 (IMAGE_QUALITY_GATE):
    IF screen_recapture OR ui_elements → CRITICAL
    ELSE IF blurry → HIGH
    ELSE IF low_quality → MEDIUM

  Rule 2 (METADATA_VERIFICATION):
    IF no EXIF timestamp → HIGH
    IF GPS present AND location mismatch → MEDIUM
    IF no GPS → LOW

  Rule 3 (REVERSE_IMAGE_SEARCH):
    IF stock_photo_likelihood = high/very_high → CRITICAL
    IF stock_photo_likelihood = medium → MEDIUM

  Rule 4 (DIGITAL_FORGERY):
    IF editing_detected → CRITICAL
    IF non_uniform_compression → HIGH
    IF watermarks_detected → HIGH

PHASE B — Vehicle & Damage Verification:
  Rule 5 (VEHICLE_MATCH):
    IF vehicle make/model doesn't match policy → CRITICAL
    IF detection confidence < 85% → MEDIUM
    IF vehicle color mismatch → MEDIUM

  Rule 6 (LICENSE_PLATE_MATCH):
    IF no plate detected → HIGH (or MEDIUM if obscured)
    IF OCR confidence < 80% → MEDIUM
    IF plate ≠ policy registration → CRITICAL

  Rule 7 (CHASE_NUMBER_MATCH):
    IF chase number present AND mismatch → HIGH

  Rule 8 (PRE_EXISTING_DAMAGE):
    IF rust OR old_repairs visible → HIGH

  Rule 10 (TOTALLED_VEHICLE_MARKERS):
    IF severity=totaled AND NO airbags/fluid/missing_parts → HIGH

PHASE C — Contextual Consistency:
  Rule 11 (NARRATIVE_CONSISTENCY):
    IF visual_evidence ≠ user_description → HIGH

  Rule 12 (MULTI_IMAGE_CONSISTENCY):
    IF plates/vehicle/lighting/damage differ across images → HIGH

PHASE D — Financial Sanity:
  Rule 13 (AMOUNT_THRESHOLD):
    IF claim_amount > ₹20,000 → MEDIUM

  Rule 14 (DAMAGE_COST_SANITY):
    IF severity=none AND claim > 0 → CRITICAL
    IF claim > AI_max_estimate × 2.0 → HIGH
    IF per-panel cost exceeds limit for severity → MEDIUM

PHASE E — Policy & History:
  Rule 15 (POLICY_ACTIVE):
    IF policy expired or cancelled → CRITICAL
    IF claim date outside policy period → CRITICAL

  Rule 16 (DUPLICATE_CLAIM_GUARD):
    IF same plate + same user claimed within 30 days → HIGH

FINAL DECISION:
  IF CRITICAL failures exist OR score ≥ 10 → REJECT
  ELSE IF score ≥ 2 → FLAG (requires human review)
  ELSE → APPROVE

Algorithm 3 — YOLOv11 Instance Segmentation (Slide 22)

MODEL: yolo11m-seg (custom fine-tuned on vehicle damage dataset)
INPUT: Damage image (RGB)
OUTPUT: List of detections with masks, class_name, confidence, bbox

DAMAGE CLASSES: Dent, Scratch, Crack, Glass shatter, Lamp broken, 
                Missing part, Rust, Deformation, Major damage

CAR PART CLASSES: Front bumper, Rear bumper, Hood, Trunk,
                  Door FL/FR/RL/RR, Fender FL/FR/RL/RR,
                  Windshield, Roof, Quarter panel, Headlight L/R,
                  Taillight L/R, License plate

PROCEDURE detect_damage_and_parts(image_path):
  results ← yolo11m_seg.predict(image_path, conf=0.15)

  damage_detections ← []
  part_detections ← []

  FOR each detection IN results:
    IF detection.class IN DAMAGE_CLASSES:
      damage_detections.append(detection)
    ELIF detection.class IN CAR_PART_CLASSES:
      part_detections.append(detection)

  // Map damage to nearest part using IoU / centroid distance
  damage_part_mapping ← []
  FOR each damage IN damage_detections:
    nearest_part ← find_closest_part(damage.bbox, part_detections)
    damage_part_mapping.append({damage_class, part_class, confidence})

  // Severity scoring
  severity_score ← calculate_severity(damage_detections)
  IF severity_score >= 0.9:  severity ← "totaled"
  ELIF severity_score >= 0.7: severity ← "severe"
  ELIF severity_score >= 0.4: severity ← "moderate"
  ELIF severity_score > 0:    severity ← "minor"
  ELSE:                        severity ← "none"

  RETURN {damage_detections, part_detections, severity, severity_score,
          damaged_panels, damage_part_mapping, summary}

Algorithm 4 — Repair Cost Estimation (Slide 23)

INPUT: damaged_panels (list of panel keys from YOLO/Groq)
USD_TO_INR ← 84.0

PART_PRICE_TABLE (USD, parts + labor industry averages):
  front_bumper: {min: 300, max: 900}
  rear_bumper:  {min: 250, max: 800}
  hood:         {min: 400, max: 1200}
  door (any):   {min: 300, max: 900}
  fender (any): {min: 200, max: 700}
  roof:         {min: 800, max: 2500}
  windshield:   {min: 200, max: 600}
  headlight:    {min: 150, max: 500}
  taillight:    {min: 100, max: 400}
  frame/chassis:{min: 1000, max: 5000}
  engine:       {min: 1500, max: 8000}

PROCEDURE estimate_repair_cost(damaged_panels):
  breakdown ← []
  seen ← empty set  // prevent duplicate parts

  FOR each panel IN damaged_panels:
    key ← resolve_panel_key(panel)  // aliases → canonical key
    IF key NOT IN seen:
      seen.add(key)
      info ← PART_PRICE_TABLE[key]
      inr_min ← info.min × USD_TO_INR
      inr_max ← info.max × USD_TO_INR
      breakdown.append({part, usd_min, usd_max, inr_min, inr_max})

  total_inr_min ← SUM(breakdown.inr_min)
  total_inr_max ← SUM(breakdown.inr_max)

  RETURN {breakdown, total_inr_min, total_inr_max}

Algorithm 5 — License Plate OCR: Groq Vision + EasyOCR Fallback (Slide 24)

PROCEDURE extract_number_plate(front_image):
  // Kerala plate format: KL DD X(X) NNNN
  //   KL = state code, DD = district (01-15, 41, 55, 63)
  //   X(X) = 1-2 series letters, NNNN = 1-4 digits

  // Step 1: Try Groq Vision (PRIMARY — Llama-4-Scout)
  IF groq_api_key available:
    base64_image ← encode_to_base64_jpeg(front_image)
    response ← groq.chat.completions.create(
      model="meta-llama/llama-4-scout-17b-16e-instruct",
      messages=[{role: "user", content: [plate_recognition_prompt, image]}],
      temperature=0.0
    )
    plate_text ← normalize(response)
    IF validate_kerala_format(plate_text):
      RETURN {plate_text, confidence: 0.95, source: "groq"}

  // Step 2: Fallback to EasyOCR pipeline
  //   2a. Detect plate region via contour analysis
  //   2b. Upscale plate crop with LANCZOS4
  //   2c. Multi-variant preprocessing (CLAHE, NLM, bilateral, adaptive threshold)
  //   2d. Run EasyOCR with letter allowlist
  //   2e. Kerala plate format correction (letter→digit, digit→letter mapping)
  //   2f. Score and rank candidates by district validity, format match

  FOR each region IN [detected_plate, bottom_third, bottom_half, full_image]:
    FOR each variant IN preprocess_variants(region):
      ocr_hits ← easyocr.readtext(variant, allowlist="A-Z0-9")
      FOR each hit: correct_kerala_plate(hit) → scored candidates

  best_candidate ← highest-scoring Kerala-format plate
  IF best_candidate found:
    RETURN {plate_text, confidence, source: "easyocr"}

  // Step 3: YOLO-assisted crop (if confidence still < 0.5)
  IF confidence < 0.5 AND YOLO_AVAILABLE:
    plate_crop ← yolo11m.find_class("license_plate", all_images)
    IF plate_crop found:
      RETRY Step 1 and Step 2 with cropped plate image

  RETURN {plate_text: None, confidence: 0}

Algorithm 6 — EXIF Metadata & Filename Parsing (Slide 25)

PROCEDURE extract_metadata(image_path):
  metadata ← {timestamp: None, gps_lat: None, gps_lon: None,
               location_name: None, camera_make: None, camera_model: None,
               filename_parsed: False}

  // Step 1: Read EXIF tags via Pillow
  img ← Image.open(image_path)
  exif_data ← img._getexif()

  IF exif_data:
    metadata.timestamp ← exif_data[DateTimeOriginal]
    IF GPSInfo in exif_data:
      lat, lon ← decode_gps(exif_data[GPSInfo])
      metadata.gps_lat ← lat
      metadata.gps_lon ← lon
      metadata.location_name ← geopy.reverse(lat, lon).address
    metadata.camera_make ← exif_data[Make]
    metadata.camera_model ← exif_data[Model]

  // Step 2: Fallback — parse timestamp from filename
  IF NOT metadata.timestamp:
    filename ← basename(image_path)
    FOR each pattern IN [PXL_YYYYMMDD_, IMG_YYYYMMDD_, IMG-YYYYMMDD-WA, ...]:
      match ← regex.match(pattern, filename)
      IF match:
        metadata.timestamp ← parse_date(match)
        metadata.filename_parsed ← True
        BREAK

  RETURN metadata

  • AI analysis is offloaded to BackgroundTasks, so claim submission returns immediately (< 300 ms) regardless of AI processing time.
  • YOLOv11 on CPU takes 5–15 seconds per image; GPU reduces this to < 2 seconds.
  • Groq API adds 3–8 seconds per claim (network latency).
  • SQLite is a limitation for concurrent writes at very high load (> 100 simultaneous writes). PostgreSQL migration is recommended for production.

Slide 38: Conclusion

AutoClaim successfully demonstrates that modern AI technologies — specifically YOLOv11 instance segmentation, Groq-powered LLaMA Vision, and EasyOCR — can be combined into a coherent, production-ready insurance claim processing system.

Key Achievements:

  • 5-stage AI pipeline that processes vehicle damage claims end-to-end: from raw photo upload to a structured APPROVE / FLAGGED / REJECT decision with full auditability.
  • 16 deterministic verification rules across 5 phases, producing a weighted fraud/authenticity score that eliminates subjective decision-making.
  • Role-based portal — policyholders, agents, and administrators each have a purpose-built interface, reducing claim processing overhead significantly.
  • Real-time forensic analysis including EXIF metadata verification, license plate OCR, AI-assisted vehicle identity, and fraudulent image detection.
  • INR-calibrated repair cost estimates based on industry-standard USD part data, converted at ₹84 per dollar.

Future Work:

  • Cloud deployment with PostgreSQL and GPU-accelerated inference on AWS
  • Mobile application (React Native) for on-the-spot claim submission from accident scenes
  • Real-time weather API integration to validate accident location/conditions against reported weather
  • Multi-language OCR support for regional language number plates
  • Expanded YOLO training dataset with Indian roads and Indian vehicle models
  • Live insurance policy integration via third-party insurance management APIs
